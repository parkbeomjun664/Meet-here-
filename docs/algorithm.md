# 중간지점 탐색 알고리즘

> 구현 위치: [`src/lib/midpoint.ts`](../src/lib/midpoint.ts)

여러 명이 모일 때 **"모두에게 가장 공평한 지점"** 을 찾는다.
단순히 좌표의 중심을 쓰지 않고, 실제 **이동시간·공평성·교통비**를 함께 평가한다.

## 전체 흐름

```
참여자 등록 (출발지 좌표 + 이동수단)
  ↓
1. 좌표 평균 → 중심점 계산
  ↓
2. 멤버 간 최대 거리 계산 → 후보 검색 전략 결정
  ↓
3. 중심점 주변 후보 지점 검색 (카카오 로컬 API, 최대 8곳)
  ↓
4. 후보 × 멤버 → 경로·시간·비용 계산 (멤버는 병렬)
  ↓
5. 세 지표를 정규화 후 가중 합산 → 점수가 가장 낮은 후보 선택
  ↓
결과 반환 (지점, 멤버별 경로, 편차/평균시간/총교통비)
```

---

## 1. 중심점 계산

```
중심 위도 = Σ(멤버 위도) / 멤버 수
중심 경도 = Σ(멤버 경도) / 멤버 수
```

이 값은 **후보를 검색할 기준점**일 뿐, 최종 답이 아니다.

## 2. 거리별 후보 검색 전략

멤버 간 최대 직선거리(Haversine)에 따라 검색 키워드와 반경을 바꾼다.

| 멤버 간 최대 거리 | 검색 키워드 | 반경 |
|---|---|---|
| 50km 초과 (장거리) | KTX역 → 고속버스터미널 → 시외버스터미널 → 지하철역 | 30~50km |
| 10~50km (중거리) | 지하철역 → 버스터미널 | 20km |
| 10km 이하 (단거리) | 지하철역 → 버스정류장 | 3~5km |

중복 제거 후 **최대 8개** 후보만 평가한다. 후보가 하나도 없으면 중심점 자체를 후보로 넣어 **항상 결과를 보장**한다.

## 3. 이동수단별 경로 계산

| 이동수단 | 1차 시도 | 폴백 |
|---|---|---|
| 대중교통 | ODsay 도시내 경로 | 40km 이상이면 도시간 추정(KTX/고속버스) → 자동차 → 거리 추정 |
| 자동차 | 카카오모빌리티 Directions | 대중교통 → 거리 추정 |
| 도보 | 직선거리 × 1.3 ÷ 4km/h | 5km 초과 시 대중교통, 40km 이상이면 도시간 추정 |

### 비용 산출
- **자동차** = 톨비(카카오모빌리티 제공) + 기름값(거리 × `170원/km`)
- **대중교통(도시내)** = ODsay 실제 요금
- **도시간(40km↑)** = KTX/고속버스 추정 (`거리 ÷ 120km/h + 접근·대기 40분`, `거리 × 180원/km`)

> ODsay는 **도시내 대중교통 위주**라 서울↔부산 같은 장거리는 경로를 반환하지 않는다.
> 그래서 임계값(40km)을 넘으면 추정 모델로 전환하고, **지도에 그릴 경로선은 카카오모빌리티의 도로 경로**를 사용한다.

폴백 체인은 **절대 실패(null)하지 않는다.** 빈 화면 대신 추정값을 보여주되, UI에는 산출 근거와 "예상"임을 함께 표시한다.

## 4. 후보 점수화 (핵심)

각 후보마다 멤버별 소요시간·요금을 모아 세 지표를 만든다.

```
공평성   = (최대시간 - 최소시간) + 표준편차
평균시간 = mean(멤버별 소요시간)
비용     = mean(멤버별 교통비)
```

**왜 표준편차를 더하나?**
`최대 - 최소`만 쓰면 한 명의 극단값에 휘둘린다. 표준편차를 함께 보면 전체 분포의 흩어짐까지 반영된다.

**왜 정규화하나?**
세 지표의 단위가 **분(minute)과 원(won)** 으로 달라, 그대로 더하면 숫자가 큰 요금이 결과를 지배한다.
그래서 후보들 사이에서 각 지표를 **0~1로 min-max 정규화**한 뒤 가중 합산한다.

```
score = 0.5 × norm(공평성)
      + 0.3 × norm(평균시간)
      + 0.2 × norm(비용)

→ score가 가장 낮은 후보를 선택
```

가중치는 `SCORE_WEIGHTS` 상수 한 곳에서 조정할 수 있다.

## 5. 성능

- 후보 × 멤버 = 최대 `8 × N` 번의 외부 API 호출이 필요하다.
- **멤버는 병렬(`Promise.all`), 후보는 순차**로 처리한다.
  전부 병렬로 돌리면 동시 요청이 `8 × N`개로 폭증해 외부 API의 요청 제한(429)에 걸린다.
  이 구조에서는 동시 요청이 최대 `N`개로 제한되면서도, 한 후보당 소요 시간은 "가장 느린 멤버 한 명" 수준으로 줄어든다.
- 멤버 간 최대 거리 계산은 모든 쌍을 비교하므로 `O(N²)`이지만, N이 작아 병목이 아니다.

## 알려진 한계

1. **실시간 교통 미반영** — ODsay는 정적 시간표 기반이라 출퇴근 혼잡이 반영되지 않는다.
2. **도보는 추정치** — 실제 도로·골목 구조를 무시하고 직선거리 × 1.3 계수만 적용한다.
3. **요금·속도 상수가 휴리스틱** — `170원/km`, `180원/km`, `120km/h` 등은 공개 요금표를 근사한 값이며 실측 검증되지 않았다. 정확도가 필요하면 공공데이터포털(TAGO) 열차·버스 API로 대체할 수 있다.
4. **중심점 주변만 탐색** — 중심에서 벗어난 교통 허브(예: 수원역)가 더 편리한 경우를 놓칠 수 있다.
5. **육상 교통만 고려** — 배·비행기는 반영하지 않는다.

## 개선 방향

- 공공데이터포털 TAGO API로 도시간 시간·요금 정확도 향상
- 후보 탐색 범위를 중심점 주변이 아닌 "교통 허브 후보군"으로 확장
- 순수 함수(`calcDistance`, `normalize`, `stdDev`) 단위 테스트 추가
